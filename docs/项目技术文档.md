## 1. 项目是什么

Kail Location 是一个 Android **位置 / 传感器模拟**应用，定位为"调试者的空间坐标实验舱"，用于在合法授权环境下做开发与测试。核心能力：

| 能力 | 说明 |
| --- | --- |
| 位置模拟 | 把设备当前定位改成指定经纬度 |
| 路线模拟 | 录制 / 回放一条路线，沿路线匀速移动 |
| 导航模拟 | 模拟导航过程，观察方向 / 状态变化 |
| 步频模拟 | 伪造计步器 / 加速度传感器数据（跑步、走路） |
| 虚拟定位 | 提供虚拟坐标供定位逻辑验证 |
| NFC 模拟 | 调试场景下的 NFC 能力 |
| WiFi / 基站模拟 | 伪造扫描到的 WiFi 热点与基站信息 |
| 沙盒 | 在 BlackBox 虚拟环境里克隆并运行第三方 App，只对沙盒内 App 生效 |
| Root 隐藏 | 对目标 App 隐藏 Root / 注入痕迹 |

> ⚠️ 合规提示：项目自带《使用条例》，明确仅限授权调试，禁止用于校园跑、打卡、网约车等。开发时请遵守仓库 `README.md` 的条款。

它是一个 **Compose + XML 混合的 MVVM 应用**，并且包含大量 **Native（C++）注入代码**和一个**独立的 Xposed 伴生模块**，因此比普通 App 复杂得多。

## 4. 模块与 Gradle 工程

`settings.gradle.kts`（根工程，名为 `location`）包含：

```kotlin
rootProject.name = "location"
include(":app")
include(":NewBlackbox:Bcore")
include(":NewBlackbox:black-reflection")
include(":NewBlackbox:compiler")
// Note: ":NewBlackbox:app" (BlackBox 的独立启动器) 故意不纳入，
// 主 App 只依赖 :NewBlackbox:Bcore
```

`KailLocationXposed/` 是**另一个独立的 Gradle 工程**（有自己的 `settings.gradle.kts`），不在根工程里，需要单独 `cd KailLocationXposed` 构建。它通过 `maven(url = "https://api.xposed.info/")` 拉取 Xposed API。

主 App `app/build.gradle.kts` 里有两段不常见的自定义构建逻辑，务必了解：

1. **Native 可执行文件改名打包**（`tasks.whenTaskAdded { ... mergeXxxNativeLibs ... }`）：
   CMake 产物里有一个名为 `kail_inject` 的**可执行文件**（ptrace 注入器）。Android 打包只接受 `lib*.so`，所以构建脚本把 `kail_inject` 复制并改名为 `libkail_inject.so` 塞进 APK 的 `lib/<abi>/`。运行时 `RootDeployer` 再把它取出来还原成可执行文件用。同理在 `stripDebugSymbols` 之后还要再补一次，避免被 strip 掉。

2. **Slim DEX 生成**（`androidComponents { onVariants { ... build<Variant>InjectSlimDex } }`）：
   注入到 `system_server` 时，需要一个尽量小的 dex（只含 `com.kail.location.inject.*` 和 `com.kail.location.lib.lhooker.*`）。脚本用 D8/R8 把这些类单独编译成**裸 dex**（非 zip 包装，因为加载器用 `InMemoryDexClassLoader(ByteBuffer)`），放进 `assets/inject.dex`。
   **为什么要这么干**：如果让 `system_server` 里的 DexClassLoader 去加载完整 33MB 的宿主 APK，Android 14 的 ART 校验经常超过 60s，触发 ptrace 看门狗把 `system_server` 杀掉。精简 dex 几毫秒就能加载完，且不需要在 `system_server` 里写 odex（SELinux 也不允许）。

## 6. 应用启动流程（入口文件 → 主界面）

### 6.1 Application 入口

**文件**：`app/src/main/java/com/kail/location/utils/GoApplication.kt`
**Manifest 声明**：`<application android:name="com.kail.location.utils.GoApplication" ...>`

`GoApplication` 是整个进程的起点，按以下顺序初始化：

```
GoApplication.attachBaseContext(base)        # 最先执行
  ├─ isMainProcess(this)                       # 判断是否主进程
  └─ BlackBoxCore.get()                         # 初始化沙盒核心
       .closeCodeInit()
       .onBeforeMainApplicationAttach(...)
       .doAttachBaseContext(this, ClientConfiguration{...})  # 配置沙盒：不隐藏Root/不开守护/不用VPN
       .onAfterMainApplicationAttach(...)
       → sandboxInitialized = true

GoApplication.onCreate()
  ├─ BlackBoxCore.get().doCreate()              # 沙盒 doCreate
  ├─ SandboxManager.init(this)                  # 沙盒管理器
  ├─ SandboxSettingsManager.init(this)
  ├─ if (!isMainProc) return                    # 非主进程到此为止
  ├─ PreferenceManager.setDefaultValues(R.xml.preferences_main)  # 读默认设置
  ├─ FirebaseApp.initializeApp(this)
  ├─ AuthManager.init(this)                     # 登录态
  ├─ UsageManager.init(this)                    # 用量限制
  ├─ registerActivityLifecycleCallbacks(this)
  ├─ RootDeployer.syncInjectLogMarkers(this)    # 把日志开关标记同步到公共目录（注入态读）
  ├─ Thread.setDefaultUncaughtExceptionHandler{...}  # 崩溃落盘到 Logs/crash_*.txt
  ├─ InjectionCrashSentinel.checkAndReport(this)  # 跨重启取证：检查上次注入是否导致设备重启
  └─ SDKInitializer.initialize(this) + setCoordType(BD09LL)  # 百度地图（用 setting_baidu_map_key）
```

> 要点：**沙盒核心必须在 `attachBaseContext` 里、宿主 Application 真正 attach 之前初始化**，否则沙盒进程无法接管类加载。这就是为什么这部分代码放在 `attachBaseContext` 而不是 `onCreate`。

### 6.2 启动入口 Activity（主壳）

**文件**：`app/src/main/java/com/kail/location/views/locationsimulation/LocationSimulationActivity.kt`
**Manifest**：`LocationSimulationActivity` 是唯一带 `LAUNCHER` intent-filter 的 Activity，即桌面图标点进来直达它。

> 注意：早期版本曾有一个 `WelcomeActivity` 权限网关页，当前代码已移除。`LocationSimulationActivity` 既是入口也是"主壳"。

它继承 `BaseActivity`，持有 `LocationSimulationViewModel`，用 `setContent { LocationSimulationScreen(...) }` 渲染主界面，并把所有抽屉导航项的点击映射成 `startActivity(...)`（见 §10）。权限申请在主界面内按需触发（`ACCESS_FINE_LOCATION` / `ACCESS_COARSE_LOCATION` 等）。

`onResume()` 里有一段**运行模式自动纠偏**逻辑：如果用户在系统"开发者选项 → 选择模拟位置应用"里选了本 App，且当前模式不是 root/xposed/sandbox，会自动把模式设为 `developer`。

## 7. 整体架构（MVVM + 前台服务 + 注入层）

本项目可以拆成**三层**：

```
┌─────────────────────────────────────────────────────────────┐
│  UI 层（Compose Activity + Screen）                           │
│  views/**  —  只负责显示和把点击事件回调给 ViewModel          │
└───────────────────────────┬─────────────────────────────────┘
                            │ StateFlow / 回调
┌───────────────────────────▼─────────────────────────────────┐
│  ViewModel 层                                                 │
│  viewmodels/**  —  持有状态、读写 SharedPreferences、         │
│                    通过 Intent 启动/控制前台服务              │
└───────────────────────────┬─────────────────────────────────┘
                            │ startForegroundService(Intent) / startService(控制指令)
┌───────────────────────────▼─────────────────────────────────┐
│  Service 层（4 种模式的前台服务）                             │
│  service/Developer  service/Root  service/Sandbox  service/Xposed │
│  —  真正干活的地方：跑位置循环、控制摇杆、发通知、下发坐标    │
└───────────────────────────┬─────────────────────────────────┘
                            │ 各模式机制不同（见 §8）
┌───────────────────────────▼─────────────────────────────────┐
│  注入 / Hook 层                                               │
│  developer: LocationManager.addTestProvider（系统标准接口）   │
│  root:      ptrace 注入 system_server + FakeLocation 钩子     │
│  sandbox:   BlackBox 的 BLocationManager                      │
│  xposed:    Xposed 模块 hook + sendExtraCommand("kail") 通信  │
└──────────────────────────────────────────────────────────────┘
```

**关键观念**：UI 几乎不直接操作定位，它只是改 ViewModel 的状态；ViewModel 用 `Intent` 启动对应模式的前台 `Service`；Service 才是各模式差异的真正实现。四种 Service 共享同一套"路线引擎 / 摇杆 / 通知 / 诊断"基础设施（见 §12）。

## 8. 四种运行模式详解

运行模式存在 `SharedPreferences` 的 `setting_run_mode` 键里，取值 `root` / `developer` / `xposed` / `sandbox`。在抽屉菜单"运行模式"对话框里切换（见 `views/common/AppDrawer.kt`）。不同模式下：抽屉里可见的菜单项不同、`getServiceClass()` 启动的 Service 类不同、底层注入机制不同。

### 8.1 Developer 模式（默认，无需 Root）

- **服务**：`service/Developer/ServiceGoDeveloper.kt`
- **机制**：用 Android 官方测试 Provider 接口 `LocationManager.addTestProvider` + `setTestProviderLocation`（封装在 `service/Developer/MockLocationProvider.kt`）。
- **前提**：需要用户在"开发者选项 → 选择模拟位置应用"里选中本 App（`GoUtils.isAllowMockLocation`）。
- **特点**：最安全、最简单，但只能改 App 自己注册的 GPS/NETWORK provider 位置，部分有反作弊的 App 能识别出 `isMock`。

### 8.2 Root 模式（需要 Root）

- **服务**：`service/Root/ServiceGoRoot.kt` + `service/Root/RootDeployer.kt`
- **机制**：综合手段，见 §13。简要说：
  1. `RootDeployer.ensureBaseline()` 把 FakeLocation 全套 `.so`、注入器 `kail_inject`、精简 dex 部署到 `/data/kail-loc/` 和 `/data/local/kail-lib/`；
  2. 用 `appops set <pkg> android:mock_location allow` 帮你自动授予模拟权限（省去开发者选项操作）；
  3. 位置主要仍走 `MockLocationProvider`（和 developer 一样的测试 provider），但 root 帮你搞定了授权；
  4. 可选地 ptrace 注入 `system_server` 注册 `service_mock_*` binder，把 WiFi / 基站 / 反检测 / Root 隐藏等能力打通；
  5. 步频模拟走注入态 `NativeStepHook`（Dobby hook libsensorservice，见 §13.6/§13.7）。
- **额外能力**：独立模拟（白名单）、Root/应用隐藏、WiFi 模拟、基站模拟。

> 代码里有一处需要注意的说明 vs 实现差异：`RootDeployer` 类注释强调"**不会**自动 ptrace 注入 system_server"（因为生产 ROM 上容易冻屏），但 `ensureBaseline()` 实际上会 `runCatching { bootstrapInjection() }` 尝试注入（带 5 秒看门狗兜底，且通过 `isSystemServerInjectionCurrent` 做开机幂等：同一开机已注入过就跳过）。阅读时以代码为准。

> 另一个保护：Root 模式启动前会过 `UsageManager.systemReadiness()` 闸门——开机不足 100 秒拒绝启动，避免 system_server 刚开机还在 JIT 时被注入卡死（见 §17）。

### 8.3 Sandbox 模式（无需 Root）

- **服务**：`service/Sandbox/ServiceGoSandbox.kt` + `service/Sandbox/SandboxLocationHook.kt`
- **机制**：基于 NewBlackbox 沙盒。在沙盒里克隆并运行第三方 App，通过 `BLocationManager.get().setGlobalLocation(BLocation(lat,lng))` 给沙盒内**所有** App 注入同一个位置（`GLOBAL_MODE`）。
- **特点**：只影响沙盒内的 App，宿主系统和沙盒外的 App 不受影响；不需要 Root。
- **入口界面**：抽屉的"沙盒"项 → `SandboxActivity`，克隆/启动用 `SandboxManager`。

### 8.4 Xposed 模式（需要 LSPosed / EdXposed 等框架）

- **服务**：`service/Xposed/ServiceGoXposed.kt`
- **伴生模块**：独立 APK `com.kail.locationxposed`（`KailLocationXposed/` 工程，见 §14）。
- **机制**：Xposed 模块在 `system_server`、`com.android.phone` 等进程里 hook 定位 / 电话 / WiFi / GNSS 方法。主 App 与模块之间**通过 `LocationManager.sendExtraCommand("kail", ...)` 这个隐蔽通道通信**（不是广播 / 文件 / socket）。
- **检测模块是否安装**：`AppDrawer.isXposedModuleInstalled()` 用 `PackageManager.getPackageInfo("com.kail.locationxposed")` 判断；没装就弹下载框。

四种模式对比：

| 维度 | Developer | Root | Sandbox | Xposed |
| --- | --- | --- | --- | --- |
| Root | 否 | 是 | 否 | 否（但要 Xposed 框架，框架本身常需 Root） |
| 影响范围 | 全系统（测试 provider） | 全系统 | 仅沙盒内 App | 全系统（hook 系统进程） |
| 反检测能力 | 弱 | 强 | 中 | 强 |
| 额外伴生件 | 无 | Native so + dex | NewBlackbox | 独立 Xposed APK |
| 服务类 | `ServiceGoDeveloper` | `ServiceGoRoot` | `ServiceGoSandbox` | `ServiceGoXposed` |

## 9. 核心调用链：点了"开始模拟"会发生什么

这是新人最该搞清楚的一条链路。以**位置模拟主界面**点击"开始"按钮为例。

### 9.1 UI → ViewModel

**界面文件**：`views/locationsimulation/LocationSimulationScreen.kt`
界面上的开始/停止按钮：

### 12.2 路线引擎

**文件**：`utils/service/RouteEngine.kt`

纯逻辑类，负责沿路线点前进。要点：
- `setupFromArray(routeArray, coordType)`：把 `[lng,lat,lng,lat,...]` 数组按坐标系转成内部 WGS84 点列，算累计距离。
- `advance(distanceMeters)`：沿当前线段推进给定米数，跨段自动跳到下一段，到终点按 `routeLoop` 决定是否回到起点。
- `seekToRatio(ratio)`：拖动进度条时跳到路线某百分比位置。
- 输出 `currentLat/currentLng/currentBea` 给服务，服务再下发。
- `progressRatio` / `buildStatusString()`：给摇杆 UI 显示"已走 / 总长"。

距离计算用 `geo/GeoMath.kt` 的 `metersPerDegLat/Lng` + 勾股近似。

**文件**：`views/joystick/JoystickWindowManager.kt`（+ `JoystickRoot/JoystickMainOverlay/JoystickAuxOverlays/FloatingNavigationControlOverlay.kt`，状态在 `viewmodels/JoystickViewModel.kt`）

### 13.2 ptrace 注入链（root 模式核心）

注入器命令格式：`kail_inject -P <进程名> -l <加载器.so> -n com.kail.location`

`inject64.cpp` 的 `injectLibraryIntoProcess(pid, lib, arg)` 流程：

```
1. PTRACE_ATTACH 目标进程 + SIGSTOP
2. 解析 /proc/<pid>/maps，识别运行时（stock Android / VMOS / Twoyi / VPhoneGaGa）
3. 用 ASLR 偏移算出远端 calloc/free/dlopen/dlerror 以及 AndroidRuntime::mJavaVM 地址
4. writeRemoteString(lib)  —— 把 .so 路径写进目标进程内存
5. callRemoteFunction(dlopen, 远端路径, RTLD_NOW)   —— 让目标进程加载我们的 .so
6. callRemoteFunction(doRun, &javaVm, "Kail.")        —— 调用 .so 导出的 doRun
7. PTRACE_DETACH
```

**看门狗**：`callRemoteFunction` 有 `gCallTimeoutMs`（默认 5 秒）。如果远端函数卡住（典型是远端 dlopen 被兄弟线程持有的 linker 锁阻塞），到时间就 `PTRACE_DETACH` 让目标进程恢复可运行，避免把 `system_server` 永久冻死（代价是本次"Inject fail"）。

每个加载器 `.so` 都导出 `doRun(JavaVM**, char*)`：
```
doRun → AttachCurrentThread → init(env)
init(env):
  - 校验 payload MD5 / 签名（fakeloc_common.h，当前默认关闭）
  - 在 /data/kail-loc/libfakeloc.so 上建类加载器
  - 反射加载 com.kail.location.inject.fakelocation.InjectDex
  - 调对应入口：init() / initZygote() / hookApplication()
```

`libfakeloc_init.cpp` 特别用 `InMemoryDexClassLoader`（ByteBuffer 加载）而不是 `DexClassLoader`，因为 `system_server` 受 SELinux 限制无法写优化 odex，DexClassLoader 会卡满看门狗窗口。

### 13.3 RootDeployer：把东西铺到设备上

**文件**：`service/Root/RootDeployer.kt`

| 常量 | 值 | 用途 |
| --- | --- | --- |
| `STAGING_DIR` | `/data/local/kail-lib` | 放注入器 + 应用内 hook so |
| `FAKELOC_DIR` | `/data/kail-loc` | 放 FakeLocation 全套 so + dex 载荷 |

| `NATIVE_HOOK_SO` | `libkail_native_hook.so` | 应用内传感器 hook |
| `INJECTOR_BIN` | `kail_inject` | ptrace 注入器 |

`ensureBaseline(context)`（每次服务启动都跑，幂等）：

其它方法：`injectAppProcess(name)`（用 apphook 注入指定 App，例如把 `com.android.phone` 拉进来 hook 基站）、`isInjectionStaged()`（检查文件是否已就位）、`revokeMockLocationAppOps()`（退出 root 模式时撤权）。

> `ensureBaselineDiagnosed(context, diag)` 是带诊断的版本，每一步都写进 `SimulationDiagnostics`，便于用户排障时一眼定位失败点。

### 13.4 InjectDex：注入成功后在目标进程里跑的 Java

**文件**：`inject/fakelocation/InjectDex.java`（被打进精简 dex）

三个反射入口对应三个加载器：

| 入口 | 由谁调 | 干什么 |
| --- | --- | --- |
| `init(context)` | `libfakeloc_init`（system_server） | 加载 `liblhooker`，注册 binder：`service_mock_location`(MockLocationManagerService)、`service_mock_wifi`、`service_mock_antidetection`、`service_hide_root`、`service_nativecatch` |
| `initZygote(startupParam)` | `libfakeloc_initzygote` | zygote 阶段加载 lhooker + `AppProcessHook.hook` |
| `hookApplication(context)` | `libfakeloc_apphook`（目标 App） | 普通 App → `AppProcessHook.applyHookToApp`；`com.android.phone` → `PhoneInterfaceManagerHook.hook`（基站） |

`hookApplication` 会按目标 App 的 `nativeLibraryDir` 判断 ABI，加载对应的 `liblhooker.so` / `liblhooker64.so` / `liblhookerx.so` / `liblhookerx64.so` 变体（支持 arm/arm64/x86/x86_64）。

这些 binder 就是 `ServiceGoRoot` 里 `IMockLocationManager` / `IMockWifiManager` / `IHideRootManager` 等 AIDL 的服务端（AIDL 定义在 `inject/fakelocation/aidl/`，服务端实现在 `inject/fakelocation/service/`）。`ServiceGoRoot` 解析到这些 binder 后，就能调 `setMockCells` / `startStepSensorMock` / `setAllowMockPackages` 等。

### 13.5 LHooker：ART 方法 Hook（自适应布局探测）

**文件**：`root/liblhooker.cpp`（C++）+ `lib/lhooker/LHooker.java` + `lib/lhooker/ArtMethodProbe.java`

原理：改写目标 `ArtMethod` 的 `entry_point_from_quick_compiled_code` 字段，让方法调用跳到生成的 RWX 蹦床（trampoline），再跳到替换方法。

**自适应探测**（重要，避免新 Android 版本写崩 system_server）：
- `ArtMethodProbe` 是一个只含 `m0()`~`m11()` 共 12 个空静态方法的探针类。ART 里同一个类声明的方法在内存中是连续排布的 `LengthPrefixedArray<ArtMethod>`，`jmethodID` 就是 `ArtMethod*`。
- Native 侧 `probeArtMethodLayout()`（`liblhooker.cpp`）解析这些方法的指针，取**两两之间最小正间距**即为当前设备真实的 `sizeof(ArtMethod)`；quick entry point 是最后一个指针大小字段，偏移 = `size - sizeof(void*)`。
- `init()` 里**优先用探测值覆盖静态表**，于是引擎对任意 ROM（含未来版本）都能自配置，不再因偏移过时而损坏 `system_server`。
- 探针类必须保持空方法体、不加构造器/字段、不被裁剪——精简 dex 用 D8 且不做 shrink，正好原样保留。

**配套诊断**：native 方法 `LHooker.nativeGetInitLog()` 把"布局探测结果 + 最终偏移 + 是否失败"汇总成文本；`LHooker.loadHookLibrary` 在 `init()` 后通过 `InjectLog` 落盘到 `/data/system/kail-loc/lhooker_init.log`，方便离线排查。

### 13.6 应用内传感器 Hook（步频，占位/降级路径）

**文件**：`native_hook/hook.cpp` + `native_hook/sensor_simulator.{h,cpp}`，JNI 封装 `root/NativeSensorHook.kt`

- `NativeSensorHook.kt` 通过 `System.loadLibrary("kail_native_hook")` 加载，声明 10 个 external 方法：`nativeSetWriteOffset/ConvertOffset/RouteSimulation/GaitParams/ReloadConfig/SetMocking/SetAuthorized/SetStepSimEnabled/Reset/InitHook`。
- `hook.cpp` 用 **Dobby** 钩两个点：
  - `hooked_send_objects`：钩 `libsensor.so` 的批量事件发送，逐个改写 0x68 字节的 sensor event。
  - `hooked_convert_to_sensor_event`：钩 `libsensorservice.so`，按节奏注入"计步器(0x13)/步检测(0x12)"事件。
- 真正的步态波形由 `gait::SensorSimulator` 单例生成（傅里叶项 / 正弦+噪声两种方案，`GaitMode` 走/跑/快跑）。
- 偏移来自磁盘上 `libsensor.so`/`libsensorservice.so` 的符号探测，或读 `/data/local/tmp/kail_sensor_offsets.txt`。

> ⚠️ 关键限制（代码注释明确）：在**控制器 App 自己的进程**里安装这个 hook 是**空操作**——因为 `libsensor.so`/`libsensorservice.so` 映射在 `system_server` 和耗传感器的 App 里，不在 `com.kail.location` 里。要让 hook 真正生效，得靠 Xposed/Zygisk 或注入态 `NativeStepHook` 把 so 加载进目标进程。所以 root 模式只 `ensureBaseline` 时步频会"优雅降级"。

### 13.7 注入态步频 NativeStepHook（root 主路径）

**文件**：`inject/utils/NativeStepHook.java`，配套 `native_hook/hook.cpp`、`sensor_simulator.{h,cpp}`

`NativeStepHook` 是**注入态**到 `libkail_native_hook.so` 的桥（与 §13.6 的应用内 `NativeSensorHook.kt` 是不同包、面向不同进程的两座桥）：

- 由 `InjectDex` 在 `system_server` 内 `System.load("/data/kail-loc/libkail_native_hook.so")` 后使用——`system_server` 才真正映射了 `libsensorservice.so`，所以合成的计步/步检测事件能进入全局传感器流。
- 用 **Dobby inline hook**（`convertToSensorEvent` / `send_objects`）而非原 `libStepSensor` 的 ELF PLT hook（后者会让 SensorService 崩溃）。
- JNI 方法：`nativeSetWriteOffset/ConvertOffset`、`nativeSetGaitParams`、`nativeSetRouteSimulation`、`nativeSetMocking`、`nativeSetStepSimEnabled`、`nativeReset`、`nativeInitHook`。
- 典型用法：`ensureLoaded()` → `install(writeOffset, convertOffset, spm)`（装 hook）→ `start(spm, mode, scheme)` / `stop()`。
- `sensor_simulator` 侧有 `RebaseStepSegment`（按节奏分段重定基，步频切换更平滑）。

> 现状理解：root 模式步频有两条路径——应用内 `NativeSensorHook`（多为占位/降级）与注入态 `NativeStepHook`（注入 `system_server` 后才真正生效）。后者是主路径。

### 13.8 注入态日志体系（InjectLog + kail_log.h）

**文件**：`inject/utils/InjectLog.java`、`cpp/kail_log.h`

注入到目标进程后运行的 Hook 代码读不到本 App 的 SharedPreferences，原来的 `log(...)` 桩方法是空的、几百处埋点全失效。现在：

- **Java/注入侧 `InjectLog`**：统一实现，带"文件:行号#方法"定位；开关用**公共目录标记文件**控制（无需 IPC / 重打包，对所有目标进程统一生效）：
  - `/sdcard/Documents/KailLocation/logs/.kail_debug` → 开启日志
  - `/sdcard/Documents/KailLocation/logs/.kail_log_file` → 额外落盘
  - `/sdcard/Documents/KailLocation/logs/.kail_verbose` → 开启高频(V)日志
  - 标记文件带 TTL 缓存（默认 5s）；W/E 始终输出，高频 V 日志限流并标注 `(+N suppressed)`；不使用 `SimpleDateFormat` 以免在 native/系统线程触发 ICU 崩溃。
- **Native 侧 `kail_log.h`**：统一日志宏 `KLOGV/KLOGD/KLOGI/KLOGW/KLOGE`，自动附加 `[file:line#func]`；高频 `KLOGV` 受 `kail::setVerboseLogging(bool)` 运行时开关控制。各 native 源已接入。
- `GoApplication.onCreate` 启动时跑 `RootDeployer.syncInjectLogMarkers(this)`，把本 App 的日志开关同步成标记文件，注入态进程就能读到。

### 13.9 反检测

**文件**：`root/libantidetect.cpp`（用 `elf_hooker.h`）

PLT/GOT 钩 libc 文件系统函数（`fopen`/`open`/`opendir` 等），在"mocking 开启"时把暴露框架自身的路径改写：含注册过的"反检测文件名"→ 乱码名；`.../maps`→`.../status`；`/su`→`/su_f`；`opendir(/sbin)`→`/etc`；`.../data/kail-loc`→`/data/_null`。JNI 面：`com.kail.location.inject.utils.LAntiDetect`。

### 14.4 状态与位置注入实现

- `xposed/core/FakeLocState.kt`：用 `AtomicReference` 存 enabled / location / speed / bearing / altitude / accuracy；`updateLocation(lat,lon)` 生成 `Location`。还负责从 `/data/local/kail-lib/libkail_native_hook.so` 加载 so（步频用）并应用 `/data/local/tmp/kail_sensor_offsets.txt` 偏移。
- `xposed/utils/FakeLoc.kt`：hook 读取的门面，holds 各 volatile 开关 + `jitterLocation()`（Ornstein-Uhlenbeck 自然抖动）/ `moveLocation()`。
- `xposed/base/BaseLocationHook.kt`：`injectLocation()` 用 FakeLoc 值造一个合成 `Location`，加抖动，按 `hideMock` 清除 mock 标记（`isMock=false`、反射清 `mMock`/`mIsFromMockProvider`），并塞伪卫星数等 extras。

## 17. 登录鉴权与用量限制

**文件**：`auth/AuthManager.kt`（登录态/token/订阅）、`auth/UsageManager.kt`（用量）。

**每次开始模拟前都会过用量闸门**（在各 ViewModel 的 start 逻辑里）：

```
UsageManager.canStartSimulation(context):   # 只检查，不扣次数
  未登录            → Toast"请先登录" → false
  已订阅            → true
  否则 RuoYiClient.checkSimulation(token):
      剩余次数 <= 0 → Toast"今日免费次数用完" → false
      否则          → true

UsageManager.consumeSimulation(context):    # 真正开始时扣一次
  已订阅 → true（不扣）
  否则 RuoYiClient.useSimulation(token)
```

所以你会在 `LocationSimulationViewModel.toggleSimulation`、`RouteSimulationViewModel.startSimulation`、`IndependentSimulationViewModel.setEnabled` 等处看到先 `canStartSimulation` 再 `consumeSimulation` 的固定套路。未登录或次数耗尽会直接 `return`，服务不会启动。

**Root 模式额外的系统就绪闸门**（`UsageManager.systemReadiness`）：开机不足 `BOOT_READY_THRESHOLD_MS`（100 秒）时拒绝启动，因为 system_server 刚开机还在大量初始化/JIT，此时 ptrace 注入会让远程函数长时间不返回，触发 watchdog 重启 system_server（设备卡死/重启）。提示文案会告知还需等待多少秒。

**改某个功能时的定位技巧**：
- 改界面/交互 → `views/<feature>/`
- 改业务逻辑/状态 → `viewmodels/<Feature>ViewModel.kt`
- 改"模拟怎么下发" → `service/<Mode>/`
- 改 Intent 常量/默认坐标 → `utils/service/ServiceConstants.kt`
- 改注入/底层 hook → `cpp/` + `inject/`

## 20. 术语表

| 术语 | 含义 |
| --- | --- |
| 运行模式 / runMode | `root`/`developer`/`sandbox`/`xposed`，存于 `setting_run_mode` |
| 前台服务 | 4 个 `ServiceGoXxx`，真正执行模拟的常驻组件 |
| 测试 Provider | `LocationManager.addTestProvider`，Android 官方模拟位置接口（developer/root 用） |
| BlackBox / 沙盒 | NewBlackbox 提供的应用虚拟化环境（sandbox 模式用） |
| ptrace 注入 | `kail_inject` 用 ptrace 把 .so 加载进目标进程（root 模式用） |
| InjectDex | 注入成功后在目标进程里运行的 Java 钩子框架入口 |
| LHooker | 自研 ART 方法 Hook 引擎（改 ArtMethod entry point，自适应探测布局） |
| ArtMethodProbe | 12 个空方法组成的探针类，运行时自测量 `sizeof(ArtMethod)` |
| Dobby / ShadowHook | inline hook 框架（传感器 hook 用 Dobby） |
| NativeSensorHook / NativeStepHook | 应用内 / 注入态两座传感器 hook JNI 桥（不同包、不同进程） |
| FakeLocState / FakeLoc | Xposed 模块里保存模拟状态 / 开关的对象 |
| sendExtraCommand("kail") | 主 App ↔ Xposed 模块的通信通道 |
| 精简 dex / inject.dex | 构建期生成、只含注入相关类的小 dex，注入 system_server 用 |
| InjectionCrashSentinel | 跨重启取证哨兵：检测上次注入是否导致设备重启 |
| SimulationDiagnostics | 每次模拟启动生成的分步诊断报告 |
| WGS84 / GCJ02 / BD09 | 三种坐标系，内部用 WGS84，百度用 BD09 |
